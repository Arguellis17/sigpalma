APORTE — Ronald
Tema: Integración de Next.js con Supabase (Auth, cookies, RLS)
Proyecto: SIG-Palma (snapshot 15-04-2026)
================================================================================

Ronald se encargó de integrar la aplicación Next.js con Supabase de forma que la
autenticación, las cookies de sesión y las consultas a Postgres sigan el patrón
recomendado para SSR: cliente de servidor creado por petición a partir de cookies,
cliente de navegador cuando hace falta interactividad (login, etc.), y sin
exponer la service role al bundle del usuario. Cableó `@supabase/ssr` en
`lib/supabase/server.ts` y `lib/supabase/client.ts`, centralizó el refresco del
JWT en `lib/supabase/middleware.ts` (`updateSession` con `supabase.auth.getClaims()`)
y lo enganchó al borde de la app mediante `proxy.ts` en la raíz del repo, de modo
que cada request pase por esa capa antes de servir rutas protegidas. Las rutas bajo
`/auth` y el flujo de login permanecen accesibles sin sesión; el resto redirige a
`/auth/login` si no hay claims válidos. Para tareas solo servidor con privilegios
elevados dejó `lib/supabase/admin.ts` usando `SUPABASE_SERVICE_ROLE_KEY`, aislado
de componentes cliente.

PASO A PASO: CÓMO SE VINCULA EL PROYECTO NEXT CON LA AUTHENTICATION DE SUPABASE
-------------------------------------------------------------------------------

1. Proyecto en Supabase Cloud. En https://supabase.com crear o abrir el proyecto
   del equipo. En Dashboard → Project Settings → API anotar la Project URL
   (formato `https://<ref>.supabase.co`) y la clave pública pensada para el
   cliente (en el panel suele aparecer como anon / publishable; es la que debe ir
   en el front, nunca la service_role).

2. Variables de entorno en el repositorio. Copiar `.env.example` a `.env.local` en
   la raíz del proyecto (Next prioriza `.env.local`). Definir exactamente:
   `NEXT_PUBLIC_SUPABASE_URL` y `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` con los
   valores del paso 1. Esos nombres coinciden con lo que importan
   `lib/supabase/server.ts`, `lib/supabase/client.ts` y `lib/supabase/middleware.ts`.
   Opcional en servidor: `SUPABASE_SERVICE_ROLE_KEY` (misma página API, clave
   service_role) solo para acciones que usan `createAdminClient()`; no usar prefijo
   `NEXT_PUBLIC_` ni subirla al repositorio.

3. Esquema de base y Auth. El login de Supabase Auth puede crear usuarios en
   `auth.users`, pero la app espera perfiles en `public.profiles`, RLS y triggers
   definidos en las migraciones del repo. Hay que aplicar el SQL canónico al
   proyecto (por ejemplo `npx supabase link` + `npx supabase db push` desde la raíz,
   o pegar la migración en el SQL Editor), según `supabase/migrations/README.md`.
   Sin eso la sesión puede existir y fallarán consultas a tablas de aplicación.

4. URLs permitidas en Supabase (Auth). En Dashboard → Authentication → URL
   configuration configurar Site URL y Redirect URLs con el origen donde corre la
   app: al menos `http://localhost:3000` para desarrollo y la URL de producción en
   Vercel cuando exista. Evita errores de redirect en flujos que dependan de
   callback URL.

5. Comportamiento en código (vínculo runtime). El formulario de login
   (`components/auth/login-form.tsx`) crea el cliente navegador con
   `createClient()` de `lib/supabase/client.ts` y llama
   `supabase.auth.signInWithPassword({ email, password })`; Supabase escribe las
   cookies de sesión en el dominio de la app. Tras éxito se hace `router.refresh()`
   para que los Server Components vuelvan a ejecutarse con cookies actualizadas.
   En cada request, `proxy.ts` invoca `updateSession`, que instancia
   `createServerClient` con lectura/escritura de cookies en la petición y ejecuta
   `getClaims()` para refrescar/validar el JWT (no usar `getSession()` en servidor
   para decisiones de seguridad, según guía Supabase SSR). Las Server Actions y
   páginas servidor que llaman `createClient()` de `lib/supabase/server.ts` reutilizan
   esas cookies y Postgres aplica RLS con `auth.uid()`.

6. Verificación. Ejecutar `npm run dev`, abrir `/auth/login`, iniciar sesión con un
   usuario existente en Authentication → Users (o el flujo de registro si está
   activo). Si la sesión es válida, el middleware deja pasar a rutas protegidas; si
   no, redirige a `/auth/login`.

7. Despliegue (Vercel u otro). Repetir el paso 2 en el panel del host: mismas
   variables `NEXT_PUBLIC_*` y `SUPABASE_SERVICE_ROLE_KEY` solo en entorno de
   servidor, y actualizar las URLs en Authentication del proyecto Supabase con el
   dominio de producción.
