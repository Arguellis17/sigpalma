APORTE — Jean Pauldi
Tema: Primer despliegue y configuración en Vercel del front
Proyecto: SIG-Palma (snapshot 15-04-2026)
================================================================================

Jean Pauldi se encargó de sacar la aplicación Next.js del entorno local y publicarla
en Vercel, conectando el repositorio Git al proyecto de hosting de modo que cada
push (o los acordados por el equipo) dispare builds y, si aplica, despliegues de
preview por rama o pull request. Ajustó el pipeline de construcción para que use
el mismo gestor de paquetes y rango de versión de Node que el repo declara o
documenta, reduciendo la brecha entre “funciona en mi máquina” y el build remoto.

La parte crítica fue configurar las variables de entorno en Vercel con los mismos
nombres que espera el código: `NEXT_PUBLIC_SUPABASE_URL` y
`NEXT_PUBLIC_SUPABASE_ANON_KEY` para el cliente que puede empaquetarse en el
navegador, y `SUPABASE_SERVICE_ROLE_KEY` únicamente en el lado servidor, sin
prefijo `NEXT_PUBLIC_`, para rutas o acciones que instancian el cliente admin
(por ejemplo creación de usuarios con privilegios). Opcionalmente dejó documentado
si hace falta `DATABASE_URL` para jobs que hablen Postgres directo; eso no forma
parte del flujo estándar del cliente Supabase en la UI.

Con ese despliegue quedó demostrado que la integración con Supabase no solo es
válida en localhost: la app es accesible por URL, los secretos quedan acotados al
runtime correcto y los fallos típicos de producción (env mal copiada, service role
filtrada al bundle) quedan acotados revisando logs de build y el panel de
variables en Vercel. Si el desarrollo local usa Webpack por estabilidad de estilos,
producción sigue las reglas del builder de Vercel; las discrepancias solo en prod
se depuran comparando esos logs con el entorno local.
