APORTE — Jhojan
Tema: PostgreSQL en Supabase, Drizzle como espejo de esquema y migraciones
Proyecto: SIG-Palma (snapshot 15-04-2026)
================================================================================

Jhojan se encargó de llevar a Supabase los esquemas que el equipo había definido
con anterioridad —en particular el trabajo de modelado y DDL inicial asociado a
Arguello— materializándolos como migraciones SQL versionadas bajo `supabase/migrations/`
(con `supabase/config.toml` en la raíz del repo) y aplicándolas al Postgres
gestionado por el proyecto Supabase Cloud. El procedimiento operativo que quedó
documentado es el siguiente: autenticarse contra la CLI con `npx supabase login`;
desde la raíz del repositorio vincular el directorio local al proyecto remoto con
`npx supabase link --project-ref <REF> --password '<contraseña postgres del panel Database>'`,
usando el mismo `<REF>` que aparece en `NEXT_PUBLIC_SUPABASE_URL`
(`https://<REF>.supabase.co`); y aplicar las migraciones pendientes al remoto con
`npx supabase db push` (opcional `--yes` en scripts para evitar confirmaciones). Como paso previo opcional, el
equipo puede levantar un Postgres local con Docker mediante `npx supabase start`
para comprobar que el SQL corre sin errores sin tocar credenciales de producción;
eso no sustituye el `db push` contra Cloud. La alternativa sin CLI es pegar el
contenido del `.sql` de migración en el SQL Editor del dashboard y ejecutarlo una
vez sobre una base que aún no tenga ese esquema. Todo ese SQL versionado es la
fuente de verdad del diseño: cualquier cambio estructural pasa por ahí antes de
considerarse “real” en la base compartida.

En paralelo configuró Drizzle (`db/schema.ts`, `drizzle.config.ts`) como espejo
declarativo del mismo esquema en TypeScript, de modo que el equipo pueda abrir
`drizzle-kit studio` (en el proyecto, expuesto como script `npm run db:studio`)
para inspeccionar tablas, documentar índices y FKs en código y, si existe
`DATABASE_URL` (cadena directa al Postgres del dashboard), ejecutar consultas o
herramientas que hablan el protocolo nativo. Dejó explícito que la aplicación en
operación normal no depende de Drizzle en cada pantalla: el runtime sigue siendo
el cliente de Supabase con RLS; Drizzle es capa de apoyo técnico y reporting, no
sustituto del cliente público.

Su aporte cerró el ciclo “definición → DDL aplicado → espejo tipado”: tras cada
aplicación de migraciones en Supabase corresponde actualizar el espejo Drizzle
para evitar desalineación entre lo desplegado y lo que el repo afirma sobre tablas
y tipos. El entorno de datos compartido para el equipo sigue siendo el proyecto
Supabase en la nube; lo probado solo en máquina local debe replicarse allí para
que todos trabajen contra el mismo esquema.
